# Degen Deal Parser — Persona Ownership Map

_Generated by Atlas 🧭 on 2026-04-17._

## Summary

Degen Deal Parser is a ~35.8K LOC FastAPI app (56 Python modules in `app/`, 16 routers, 44 Jinja templates, 27 test files, 3.4K LOC of PowerShell) running in production at `ops.degencollectibles.com`. Two product verticals dominate: **Discord deal parsing** (raw messages → stitched → parsed → normalized transactions) and **TikTok Shop livestream ops** (orders, streamer dashboard, analytics). Risk concentrates in three places — (1) the parser/stitcher/worker stack (determinism is a stated invariant and the prior audits have flagged three correctness bugs), (2) the TikTok integration (7K LOC, webhook signing, token refresh, unindexed hot path), and (3) deploy/ops hygiene (DB creds in repo, no rollback, `/health` lies when the DB is dead). `app/shared.py` at 5K LOC is a god-module and the riskiest refactor target.

**Routing strategy:** Forge is the default for anything that edits files. Atlas front-loads planning for any change touching ≥3 files or crossing area boundaries. Echo gates merges on parser/auth/financial code. Scout owns live Q&A, log grep, and data lookups. Beacon owns continuous watches. Main = Jeffrey for customer-facing or scope calls.

## Routing heuristics

| If the task is… | Route to… | Why |
|---|---|---|
| Bug fix with clear repro + known file | **Forge** | Direct edit + PR |
| "Why did X happen in prod?" / log grep | **Scout** | Fast log/DB investigation, no writes |
| Data question ("how many X this week?") | **Scout** | Quick read-only SQL |
| Pre-merge security/correctness review | **Echo** | Skeptical eye before deploy |
| Post-incident audit | **Echo** | Writes critique, not patches |
| "Should we build X?" / design tradeoff | **Atlas** | Spec + tradeoffs |
| Cross-cutting refactor (>3 files, >1 area) | **Atlas → Forge** | Plan first, then build |
| Live watch ("ping me when X happens") | **Beacon** | Long-running monitor |
| Cron/scheduled job | **Beacon** | Owns recurring |
| Customer-facing / business scope call | **Main (Jeffrey)** | Owner decides |
| Anything touching `shared.py` | **Atlas → Forge → Echo** | 5K LOC god-module; plan, build, review |
| Anything touching parser/stitcher | **Forge → Echo** | Determinism is a stated invariant |
| Anything touching auth/webhook signing | **Forge → Echo** | Security-sensitive; second pair of eyes mandatory |

## How to keep Forge from being a bottleneck

Forge owns the primary slot on almost every area because Forge is the only persona that writes code. To prevent Forge queue overload:

- **Route reads to Scout.** "Show me X", "how many X this week?", "why did X happen?" — all go to Scout. No file edits = not Forge's job. This is the biggest single deflection.
- **Gate with Echo before merge, not after.** For auth, webhook signing, parser, and financial code, make Echo review mandatory *before* merge. Prevents Forge from having to cycle back on bugs Echo would have caught.
- **Force Atlas on multi-area changes.** Anything touching ≥3 files or crossing area boundaries (e.g., parser + models + reporting) gets an Atlas spec first. Prevents Forge from doing half-a-plan-while-coding and hitting a dead end.
- **Beacon owns the "is it still working?" loop.** Jeffrey doesn't need Forge to check if the deploy stuck — Beacon can watch `/health` and page if anything breaks.
- **Main (Jeffrey) owns scope calls.** "Should we build Z?" and "is this the right tradeoff for the business?" don't go to Forge. They go to Jeffrey, optionally with Atlas as design partner.
- **Micro-tasks (<30m) bypass Atlas.** Don't plan a 30-minute index add. Straight to Forge.

Rough target distribution (% of incoming tasks): Forge 55% · Scout 20% · Atlas 10% · Echo 10% · Beacon 3% · Main 2%.
